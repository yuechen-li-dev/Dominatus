# Dominatus

Dominatus is an early-stage .NET 8 AI runtime kernel focused on **deterministic orchestration**. Today, it combines:

- A step-based behavior model authored as `IEnumerator<AiStep>` nodes.
- A hierarchical finite-state machine (HFSM) runtime to compose and execute those nodes.
- Utility decision policies for stable switching between options.
- A typed actuation/event model for command dispatch and completion.
- Checkpoint + replay primitives for auditable execution.

This README intentionally describes only what appears to be true in the current codebase.

## What Dominatus is (current, factual scope)

At runtime, Dominatus executes AI behavior as **iterator-driven step streams** (`AiStep`) through a `NodeRunner`, and coordinates those behaviors in an HFSM stack managed by `HfsmInstance`.

Key properties currently implemented:

- **Iterator-first node model**: `AiNode` is a delegate `IEnumerator<AiStep> AiNode(AiCtx ctx)`.
- **One-step intent protocol**: each `yield return` emits a typed step such as wait, transition, event wait, actuation, or decision.
- **HFSM composition**: states are represented in `HfsmGraph` and executed as active frames with root/stack semantics.
- **Utility decisions**: `Decide` + `DecisionPolicy` support hysteresis, min-commit windows, and tie handling.
- **Actuation ABI**: commands implement `IActuationCommand` and are dispatched through `IAiActuator`/`ActuatorHost` with immediate or deferred completion.
- **Replay/checkpoint foundation**: persistence models capture blackboard and in-flight actuation context, with replay events and a `ReplayDriver` to reinject nondeterministic inputs.

## Core architecture (as implemented)

### 1) Node execution (`NodeRunner`)

`NodeRunner` drives the enumerator lifecycle (`Enter`, `Tick`, `Exit`) and interprets yielded steps:

- Wait steps (`WaitSeconds`, `WaitUntil`) stall progress until conditions pass.
- Event waits (`IWaitEvent`) consume from per-agent event buses with cursors.
- Actuation dispatch (`Act`) can store ids in blackboard and publish completion events.
- Completion/control steps (`Goto`, `Push`, `Pop`, `Succeed`, `Fail`) are emitted upward for HFSM handling.

Natural end of an iterator is treated as success; exceptions become failure.

### 2) HFSM orchestration (`HfsmInstance`)

`HfsmInstance` owns active state frames and:

- Initializes and restores active paths.
- Applies transition/interrupt scans with cadence and dirty-key dependency filtering.
- Evaluates utility `Decide` steps via decision memory to prevent unstable thrashing.
- Applies structural state changes while preserving root-frame behavior when configured.

### 3) OptFlow authoring surfaces

- `Dominatus.OptFlow.Ai` provides concise builders (`Ai.Wait`, `Ai.Push`, `Ai.Decide`, `Ai.Act`, `Ai.Await`, etc.).
- `Ariadne.OptFlow` provides dialogue-oriented steps.
- `Dominatus.Llm.OptFlow.llm` adds lightweight helpers for LLM-style tool envelopes and await semantics.
- `Dominatus.UtilityLite.Utility` provides reusable utility scoring composition helpers.

### 4) Runtime and persistence

- `AiWorld` + `AiAgent` provide execution context (clock, mailbox, event bus, blackboard, actuator).
- `ActuatorHost` supports handler registration and immediate/deferred completion publication.
- Persistence includes checkpoint structures and replay logs/events used by `ReplayDriver`.

## Unique approach (credible differentiators today)

Dominatus' strongest differentiator right now is its bias toward **determinism and auditability** rather than purely model-driven orchestration:

- Behavior is explicit and inspectable in typed steps.
- Tool/command execution is integrated as typed actuation contracts.
- Replay mechanics are built into core architecture, not bolted on.
- HFSM + utility decisions provide predictable control flow under bounded policies.

This does not yet make Dominatus “complete” as a full AI foundation, but it is a meaningful architectural direction.

## Writing nodes with `IEnumerator<AiStep>`

### Minimal node

```csharp
using Dominatus.Core.Nodes;
using Dominatus.Core.Nodes.Steps;
using Dominatus.Core.Runtime;
using Dominatus.Core.Runtime.Commands;
using Dominatus.OptFlow;

static IEnumerator<AiStep> Idle(AiCtx ctx)
{
    yield return Ai.Wait(0.10f);

    if (ctx.World.Clock.Time > 5f)
        yield return Ai.Succeed("idle-complete");
    else
        yield return Ai.Wait(0.25f);
}
```

static IEnumerator<AiStep> DoWork(AiCtx ctx)
{
    yield return Ai.Act(new LogCommand("hello"), Keys.LastCmd);
    yield return Ai.Await(Keys.LastCmd, Keys.LastResult);
    yield return Ai.Succeed("work-complete");

## Claims that are accurate right now

- Dominatus uses CLR iterator state machines (`IEnumerator<AiStep>`) as foundational behavior units.
- Those units are composed by an HFSM runtime and can include utility decision steps.
- The repository includes an LLM-oriented OptFlow helper package (`Dominatus.Llm.OptFlow`) for tool envelope dispatch over the same actuation pathway.
- Replay/checkpoint capabilities exist and are treated as first-class concerns in core runtime design.

## Gaps between current state and ambition

If the ambition is a next-generation AI agent foundation, the current code suggests these major gaps:

1. **First-class LLM runtime package is minimal**  
   LLM support is currently helper-centric (`llm` DSL + `LlmToolCall` envelope) rather than a complete model runtime abstraction.

2. **No robust tool schema/manifest registry**  
   Tool discoverability, schema validation, and strongly governed dispatch contracts are not yet a full subsystem.

3. **Replay lacks explicit model transcript semantics**  
   Replay events capture several nondeterministic host inputs; model request/response transcript logging is not yet clearly represented as first-class replay events.

4. **Safety/policy stack is still lightweight**  
   Policy interfaces exist, but richer governance layers (risk tiers, approvals, context-aware guards) are not yet fully realized.

5. **Memory architecture is mostly blackboard-centric**  
   There is solid runtime state and checkpointing, but no explicit broader long-horizon memory subsystem.

## Practical takeaway

Dominatus appears to be a promising deterministic orchestration kernel with clear architectural intent. Its current strengths are in **runtime determinism, step semantics, HFSM control, and replayability**. The path to “next-generation foundation” most likely depends on deepening LLM runtime contracts, tool governance, and memory systems while preserving its deterministic core.
